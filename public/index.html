Telegram 機器人設定說明

如何建立 Telegram 機器人並取得 Token？

  1. 在 Telegram 中搜尋官方帳號 @BotFather 並開始對話。
  2. 輸入指令 /newbot，依指示設定機器人名稱，即可取得 Bot Token。

如何取得 Chat ID？

  1. 將您建立的機器人加入您的群組，或與機器人開啟個人對話。
  2. 在該對話中隨意發送一則訊息。
  3. 存取以下網址（需替換為您的 Token）：
    https://api.telegram.org/bot[YOUR_TOKEN]/getUpdates
    找到 "chat":{"id":...}，該數字即為 Chat ID（群組 ID 通常是負數，例如 `-10012345`）。

注意：必須先與機器人開啟對話，否則發送會失敗。

關於 AI GATE 監控系統

AI GATE 智慧車牌監控與出入管理系統，整合了 YOLOv8 深度學習偵測演算法與 PaddleOCR 文字辨識管道，可在純 CPU（無 GPU/CUDA 加速）硬體環境下，實現雙向車道、無盲區的高精確度動態追蹤。

系統採用了影像移動偵測（Motion Detection）與高斯 Sobel 邊緣紋理分析，結合智慧排程，在提供高可靠性安全監控的同時，最大限度地節省伺服器硬體開銷。

核心特色與技術亮點

雙相機無盲區監控

大門與中庭雙相機感應範圍放寬至全幅，覆蓋進出校雙向所有車道（包括極左與極右車道），徹底解決摩托車與偏向行駛漏檢問題。

車牌影像前處理增強管道

對車牌小圖進行高度拉伸（cv2.INTER_CUBIC）、自適應直方圖等化（CLAHE 對比增強）與 Unsharp Mask 高斯銳化，使夜間、逆光與小型機車車牌的辨識信心度大幅提升。

22/5 智慧門禁偵測排程

每日 05:00~22:00 自動處於監控啟動狀態，避開日間反光；22:00~翌日 05:00 啟用實際大門狀態偵測，關門時自動休眠（暫停 LPR 辨識以降低 CPU 負載），兼顧便利與節能。

雙相機獨立啟用控制

支援大門與中庭相機獨立啟用、停用及「全部啟用/停用」批次控制，前端影格自適應分流渲染，並整合了多執行緒防卡死重連與 SQLite WAL 併發優化。

HTTP Basic Auth 安全防護

全系統所有靜態網頁、API 介面及即時串流影像均受安全性驗證保護，使用者亦可在系統設定中隨時進行管理員帳密修改。

Telegram 警報與即時推播

當偵測到黑名單內追蹤車牌、或特定事件觸發時，系統會立即向您的 Telegram 群組發送即時通知，並自動附帶車牌採證小圖與完整現場大圖。

標準中介流代理 (go2rtc)

全面導入 go2rtc (v1.9.14) 代理架構，原生支援 TCP/UDP 自動重連與 NVR 138 特殊視訊流的 FFmpeg 高效拉流，保障全天候穩定的即時串流。

多邊形閘門區域與 ByteTrack 追蹤

整合 Supervision 閘門偵測區（PolygonZone）與線上編輯器，搭配 Kalman 濾波動態追蹤（ByteTrack），可手動任意拖曳頂點、即時熱載入生效，並支援雙車道多車並行獨立辨識。

v3.4 2026-07-13 (當前版本)

多邊形閘門偵測編輯器與 Supervision ByteTrack 追蹤整合

  • 多邊形偵測區域編輯器：新增「偵測區域設定」面板，支援滑鼠與觸控拖曳四點頂點，直覺調整 `sv.PolygonZone` 閘門偵測區過濾範圍，實時換算 1920x1080 全畫幅像素座標。
  • 熱重載免重啟機制：後端新增 `/api/zones` API，區域變更儲存後直接寫入 `zone_config.json` 並即時更新記憶體過濾遮罩，無須重啟 LPR 引擎。
  • Supervision ByteTrack 追蹤整合：引進 `sv.ByteTrack` 追蹤演算法（Kalman 濾波與 IoU 配對）全面取代舊有單幀 IoU 比對，確保在 30 幀的觸發窗口期中維持車輛 ID 穩定度，大幅提升移動中車輛的追蹤可靠性。
  • 多車並行偵測與去重：利用 ByteTrack ID 分組處理，當多台車輛同時通過時，可各自獨立積累最佳車牌並寫入資料庫，完全解決過去一次只能記錄一台車的限制。
v3.3 2026-07-07

CPU負載極致優化與 OpenVINO 運作穩定度提升

  • OpenVINO CPU 穩定加速：將 YOLO 辨識轉為 OpenVINO 模型並固定於 CPU 執行，避開 Intel UHD 630 顯卡驅動的 CISA 編譯器閃退 Bug，同時仍保有 27ms 的高速推論效能。
  • 背景運算執行緒空閒休眠：導入 `KMP_BLOCKTIME=0` 與被動等待策略，強制 OpenMP/TBB 等運算庫在空閒時立即釋放 CPU 並休眠，解決高達 45% 的背景 CPU 空轉消耗。
  • 運動偵測防噪與連續幀過濾：大門與大門002的 driveway 像素門檻分別提高至 450/350，並加入「連續 2 幀運動判定」，100% 過濾風吹落葉與 I-frame 訊號刷新的無效觸發，徹底消除冷卻期造成的機車漏報盲區。
  • 影像處理頻率平衡：解碼睡眠時間微調至 80ms 並跳過 1/2 影格（有效處理率 6fps），將 YOLO 呼叫間隔拉長至 3 幀，達成高捕捉率與超低 CPU 使用率（僅 3~4%）的完美平衡。
v3.2 2026-07-05

機車偵測高靈敏度優化與全網頁抗閃爍重構

  • 網頁全介面抗閃爍重構：關閉 Chart.js 圖表更新動畫，側邊欄車流清單改用 DOM diffing 增量前插更新，防止全部車牌截圖重複載入閃爍。
